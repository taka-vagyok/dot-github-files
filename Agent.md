# AI Agent Instructions (Agent.md)

このファイルは、本リポジトリで作業するGitHub Copilot（およびその他のAIエージェント）のためのガイドラインです。コードの生成、リファクタリング、テストの作成を行う際は、必ず以下のルールとコンテキストに従ってください。

## 1. プロジェクトの概要 (Project Context)
* **プロジェクト名**: dot-github-files
* **目的**: GitHub Copilot 向けのサンプルの Prompt（プロンプト）、Agent（エージェント）、および Skill（スキル）を定義し、AI開発のベストプラクティスに基づいた効果的な実装例を提供すること。
* **主要なドメイン**: AIエージェントのオーケストレーションと、再利用可能なスキルの構築。
* **【重要】リポジトリの性質**: 本リポジトリは、実際の開発プロジェクトにおいて `.github/` 配下に配置される設定ファイル群を管理するテンプレート・プロジェクトです。

## 2. 【最重要】AIエージェントへの強い指示: 実装前の思考プロセス (Design Thinking Process)
ユーザーから「〇〇の機能を作って」「〇〇のスキル/プロンプトを作って」と依頼された場合、**いきなり `.agent.md` や `SKILL.md` を作成・修正してはいけません。**
一つの作業（タスク）は、必ず「Prompt」「Orchestrator Agent」「Specialist Agent」「Skill」の組み合わせで構成されるべきです。「神エージェント（全部入り）」や「神スキル（推論を含む）」を作ることは厳禁です。

**必ず以下のプロセスに従って要件を分離し、計画（ワークフロー）を立ててから実装してください。**

1. **要件の分解 (Requirement Analysis)**
   - ユーザーの要求を「入力を受け付ける部分」「進行を管理しレビューする部分」「具体的な推論や判断を行う部分」「外部システムやファイルに触れる物理的な作業部分」に分解する。
2. **コンポーネントへの割り当て (Component Mapping)**
   - **Prompt (`prompts/`)**: ユーザーと対話し、入力を整理してAgentに渡すだけの「入り口」はどれか？
   - **Orchestrator Agent (`agents/`)**: 処理手順を計画し、サブエージェントを呼び出し、結果をレビューして反復ループを回す「監督」の役割は何か？（直接作業はしない）
   - **Specialist Agent (`agents/`)**: 特定のドメイン知識に基づき、推論や判定を行う「専門家」の役割は何か？
   - **Skill (`skills/`)**: 実際にコマンドを実行する、ファイルを操作する、APIを叩くといった「手足（副作用）」の役割は何か？（ディレクトリとスクリプトで構成）
3. **設計の確認と実装 (Design Review & Implementation)**
   - 上記の分離によって、一つのファイルに処理が集中していないか確認する。
   - 分離した設計に基づいて、初めて各仕様書（`spec/*.md`）に従いファイル群を生成・修正する。

## 3. ディレクトリ構成とファイル規約 (Architecture & Conventions)
前述の通り、本リポジトリのルートディレクトリが実質的な `.github/` ディレクトリに相当します。ファイルを追加する際は、以下の構成に従い、Markdown形式（フロントマター付き）で記述してください。

* `prompts/`: ユーザーの開始点となるプロンプトファイル（拡張子: `.prompt.md`）。複雑なロジックは持たせないこと。
* `agents/`: 特定の役割を持つ自律的な作業者の定義ファイル（拡張子: `.agent.md`）。
* `skills/`: エージェントが手足として使う具体的なツールの実装。
* `spec/`: 各コンポーネントの詳細な作成ルールを定めた仕様書（`agent-spec.md`, `prompt-spec.md`, `skill-spec.md`）。

**命名規則**: 関連する作業をまとめる際は、対象ドメインやプロジェクトを示す接頭辞を付与すること（例: `[prefix]-[name].agent.md`）。

## 4. エージェント設計のベストプラクティス (Agent Design Rules) ★最重要
新しいエージェント（`.agent.md`）を設計・生成する際は、以下の厳格なルールに従うこと。

1. **ツールの限定 (Prevent Tool Overload)**: フロントマターの `tools` セクションで、エージェントが利用可能なツールを必要最低限に限定すること。不要なツールはコンテキストを消費し、AIの判断を曇らせる原因となる。
2. **MCPよりCLIを優先 (Prevent Context Rot)**: コンテキスト消費を抑えるため、可能な限りMCPツールよりも軽量なCLIコマンド（`gh`, `grep`, `jq` など）の利用を優先して指示すること。
3. **手順の明示**: 複雑なタスクを指示する場合は、`#tool:todo` を併用して「やることリスト」を明示し、エージェントの行動をガイドすること。

## 5. オーケストレーターパターンの実装 (Orchestration & Subagents)
オーケストレーターとなるメインエージェントを生成・修正する際は、以下のルールを守ること。

* **直接編集の禁止**: オーケストレーターには `edit` などの直接コードを編集するツールを与えないこと。ワークフローの進行管理（指揮）のみに専念させる。
* **サブエージェントの明示的呼び出し**: タスクの委譲には必ず `agent/runSubagent` ツールを使用し、呼び出しパラメータとして `prompt` と `agentName`（カスタムエージェント名）を明示すること。
* **イテレーションの強制**: オーケストレーターのタスク手順には「専門サブエージェントの成果物を事前に定義したテスト/仕様と照らし合わせてレビューし、基準を満たさなければ再実行させる『反復ループ』」を必ず組み込むこと。

## 6. テストと品質保証 (Testing & QA)
AIエージェントは、本リポジトリにおいて**テスト駆動開発（TDD）**の原則に従うこと。
* 新しい Agent や Skill の実装を要求された場合、**コードを書く前に必ずテストコード（Spec/Acceptance Criteria）を先に作成・提案**すること。
* 推測でコードを書かないこと。要件が不明確な場合は、実装を始める前に人間に質問すること。
