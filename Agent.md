# Agent.md

本リポジトリは GitHub Copilot 向けのサンプルとなる Prompt、Agent、および Skill を定義・提供しています。
各要素の作り分け方、および命名規則などのベストプラクティスについて以下に説明します。

## なぜ Prompt、Agent、Skill に分割し、オーケストレーターパターンを採用するのか？

「Prompt」「Agent」「Skill」に分割し、オーケストレーターに「テスト駆動（TDD）」や「反復（イテレーション）」を組み込むような少し複雑な設計にするのか、その本質を理解することは非常に重要です。
このアーキテクチャ（オーケストレーターパターンおよび責務の分離）が優れている理由は、主に以下の3点に集約されます。

### 1. 「プロンプトのスパゲッティ化」を防ぎ、保守性を高めるため
1つの巨大なプロンプトに「システムへの挨拶」「要件定義のやり方」「コードの書き方」「ツールの使い方」「出力形式」をすべて詰め込むと、いわゆる**「神プロンプト（巨大で複雑なプロンプト）」**が出来上がります。
*   **課題**: どこかを修正すると別の挙動がおかしくなる（デグレが起きる）、新しい機能を追加したくても文字数（コンテキスト）の限界が来る、という問題に直面します。
*   **解決策**: コンポーネントを分割（モジュール化）することで、「APIの仕様が変わったからSkillだけ修正しよう」「新しい作業が必要だからAgentを1つ追加しよう」といった形で、人間が管理・保守しやすいシステムになります。

### 2. LLMの「認知負荷」を下げ、ハルシネーション（幻覚）を防ぐため
LLM（大規模言語モデル）は、一度に大量の指示や多様な役割を与えられると、注意力が散漫になり（Attentionの希薄化）、指示を無視したり、辻褄の合わない回答を生成しやすくなります。
*   **解決策**: 「あなたは〇〇の専門家です」と単一の役割（単一責任の原則）を与えられたエージェントは、そのタスクにおいて極めて高いパフォーマンスを発揮します。思考を分散させないことで、出力の精度と安定性が飛躍的に向上します。

### 3. 「やりっぱなし」を防ぎ、自律的な品質保証（担保）を行うため
LLMに単発でタスクを依頼すると、一見それっぽいですが抜け漏れやバグのある「60点の成果物」を出して終わることがよくあります。
*   **解決策**: オーケストレーターに「事前にテスト（受入条件）を作らせる（SDD/TDD）」「成果物をテストと照らし合わせてレビューし、ダメなら再実行させる（反復ループ）」というプロセスを強制することで、人間が手動でリテイクを指示しなくても、AI同士が自律的に品質を80点、90点へと引き上げてから最終結果を出力するようになります。

## プロンプト、エージェント、スキルの作り分けについて

オーケストレーターパターンに基づく自律型AIエージェントシステムでは、各コンポーネントの「責務の分離（単一責任の原則）」が成功の鍵となります。以下にそれぞれの役割を定義します。

*   **Prompt (プロンプト)**:
    ユーザーとシステムの接点であり、ワークフローの「開始ボタン（エントリーポイント）」です。ユーザーからの直接の入力、または外部システムからのトリガーを受け取る最前線のインターフェースとして機能します。
*   **Agent (エージェント)**:
    特定の「役割（Role）」と「自律的な思考・判断能力」を持った作業者です。進行管理を行う「オーケストレーター」と、実作業を行う「専門エージェント」に明確に分割して利用します。
*   **Skill (スキル)**:
    エージェントが手足として使う「ツール（API、検索、スクリプト実行、データベース操作など）」です。エージェントの推論能力だけでは解決できない、外部情報の取得やシステムへの副作用（書き込み等）を伴う処理を担当します。

## 命名規則のベストプラクティス

一連の処理や関連する作業をまとめる際、それらを表す接頭単語（プレフィックス）をつけることをルールとします。これにより、機能がどのタスクに属しているか明確になります。

*   **特定の作業に関連する場合**:
    その作業を表す単語を区切り文字（`.` または `-`）で区切って接頭辞とします。
    例：`setup-environment`, `deploy.frontend`
*   **特定の作業に依存しない共通機能の場合**:
    `common` などの汎用的な単語を接頭辞とします。
    例：`common-logger`, `common.utils`
