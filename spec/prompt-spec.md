# Prompt Spec

本ドキュメントは、GitHub Copilot 向けのプロンプト (Prompt) を作成する際のルールとベストプラクティスを定義したものです。
プロンプトはユーザーとシステムの接点であり、ワークフローの「開始ボタン（エントリーポイント）」として機能します。

## 1. 責務の分離と委譲

*   **ロジックの排除**:
    プロンプト内に複雑な条件分岐、推論、作業手順を記述してはいけません。推論や作業はエージェントの責務です。
*   **委譲に徹する**:
    「ユーザーからの入力データ（Context）」を整理し、「どのエージェント（主にオーケストレーター）にタスクを委譲するか」のみを定義することに集中してください。

## 2. 役割と制約の明確化

*   **ペルソナ (Persona) の設定**:
    システムプロンプトの冒頭で、AIに演じさせる役割を明確に定義してください。
*   **制約条件 (Constraints) の明記**:
    タスク実行時に守るべきルール（例: 「必ず日本語で回答する」「説明は簡潔に行い、コードのみを出力する」）を箇条書きで明確に記述します。

## 3. 構造化と文脈の提供

*   **区切り文字 (Delimiters) の使用**:
    トリプルクォート (`"""`) やバッククォート (```)、XMLタグ (`<context>...</context>`) などを活用し、指示内容、文脈、参照データ、ユーザー入力を明確に区別してください。
*   **十分なコンテキストの提供**:
    タスクの背景や目的、使用しているフレームワーク・ライブラリのバージョンなど、回答を導き出すために必要な情報を過不足なく提供します。

## 4. 回答精度の向上

*   **Few-Shot Prompting (例示)**:
    期待する入力と出力のペアをいくつか提示することで、AIの回答フォーマットと精度を劇的に向上させることができます。
*   **Chain of Thought (思考プロセスの誘導)**:
    複雑なタスクでは、「ステップ・バイ・ステップで考えてください (Think step-by-step)」といった指示を含め、AIに推論過程を出力させることで、論理的なエラーを防ぎます。
*   **説明 (Description) への発火条件の明記**:
    プロンプトの `description` には、どのような状況やユーザーの意図で呼び出すべきか明確な発火条件を記述してください。

## 5. レビューチェックリスト (Review Checklist)

AIコンポーネントをレビューするエージェント（例: `ai-component-manager.reviewer.agent`）は、プロンプトファイルが以下の基準を満たしているか確認してください。

*   [ ] **拡張子と配置**: `prompts/` ディレクトリ配下にあり、拡張子が `.prompt.md` であるか。
*   [ ] **フロントマター**: YAML形式で `name` と、明確な発火条件を含む `description` が定義されているか。
*   [ ] **ロジックの排除**: プロンプト内に条件分岐、複雑な作業手順、推論ロジックが書かれておらず、単なる「入り口」として機能しているか（神プロンプトになっていないか）。
*   [ ] **委譲の徹底**: 整理した入力情報を、必ず `agent/runSubagent` ツール経由で適切な「オーケストレーターエージェント（`.agent.md`）」に渡すよう指示しているか。

## 6. プロンプトのサンプルフォーマット

プロンプトを作成する際は、以下のMarkdownフロントマター形式（拡張子: `.prompt.md`）を基本としてください。

```markdown
---
name: "my-task.prompt"
description: "ユーザーが〇〇について質問した際や、〇〇のタスクを開始したい場合に呼び出されるエントリーポイントです。"
---

あなたは経験豊富なシニアソフトウェアエンジニアです。
ユーザーからの入力をもとに、解決に必要な要件を整理し、適切なエージェントにタスクを委譲してください。

【制約条件】
- 複雑な推論や作業の実行は行わないこと。
- 整理した要件に基づき、必ずオーケストレーターエージェント（`agent/runSubagent`経由）を呼び出すこと。
- 出力は日本語で行うこと。
```
