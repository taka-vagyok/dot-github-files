# Skill Spec

本ドキュメントは、GitHub Copilot 向けのエージェントが実行するスキル (Skill/Tool) を作成する際のルールとベストプラクティスを定義したものです。
スキルはエージェントの推論能力だけでは解決できない、外部情報の取得やシステムへの副作用（書き込み等）を伴う処理を行うための「手足」となるツールです。

最新の仕様は [Agent Skills Specification](https://agentskills.io/specification) に準拠します。

## 1. ディレクトリ構成とファイル規約

スキルは単一のファイルではなく、ディレクトリとして構成されます。最低限 `SKILL.md` が必要です。

```text
skill-name/
├── SKILL.md          # 必須: メタデータ(フロントマター) + 指示内容
├── scripts/          # 任意: 実行可能なコード(Python, Bash等)
├── references/       # 任意: エージェントが読み込める追加のドキュメント
└── assets/           # 任意: 静的リソース(テンプレート、画像など)
```

## 2. SKILL.md のフロントマター (メタデータ)

`SKILL.md` の先頭には YAML 形式のフロントマターが必須です。

*   **name** (必須): 1-64文字。小文字の英数字とハイフンのみ。ハイフンで始まったり終わったりしてはいけません。ディレクトリ名と一致させる必要があります。
*   **description** (必須): 1-1024文字。「何を行うか」と「いつ使用すべきか（発火条件）」を具体的に記述します。
    *   *悪い例*: 「ウェブを検索します」
    *   *良い例*: 「ユーザーの入力に〇〇に関する要件が含まれている場合や、最新の情報を確認する必要がある場合は、【必ず】このスキルを使用してください。」
*   **license** (任意): ライセンス名。
*   **compatibility** (任意): 動作環境の要件（例: `Requires Python 3.14+`）。
*   **metadata** (任意): 任意のキー・バリューのメタデータ。
*   **allowed-tools** (任意): 使用を許可するツールのリスト（スペース区切り）。

## 3. ボディコンテンツと段階的開示 (Progressive Disclosure)

フロントマターの後に続くMarkdownボディには、エージェントがタスクを効果的に実行するための手順、入出力の例、エッジケースのハンドリングなどを記述します。

*   **コンテキストの節約**: メインの `SKILL.md` は500行未満に抑えることが推奨されます。
*   **ファイル参照**: 長い指示や詳細なリファレンスは `references/REFERENCE.md` などに分割し、`SKILL.md` 内から相対パスで参照させてください。これにより、エージェントは必要な時だけ詳細情報をコンテキストに読み込みます。

## 4. 入出力とエラーハンドリング

*   **入出力の定義**: 必要な引数や期待される出力フォーマットがある場合は、Markdownボディ内で明確に定義してください。
*   **手順の具体化**: スクリプト(`scripts/`)を実行する手順や、例外発生時の振る舞いを明確に定義することで、システムの堅牢性を高めます。
*   **明確なエラーメッセージ**: スクリプト等が失敗した場合は、自己修復やリトライが行えるよう、具体的にエラー原因を示すメッセージを出力させてください。

## 5. レビューチェックリスト (Review Checklist)

AIコンポーネントをレビューするエージェント（例: `ai-component-manager.reviewer.agent`）は、スキルコンポーネントが以下の基準を満たしているか確認してください。

*   [ ] **ディレクトリ構造**: スキルが単一のファイルではなく、ディレクトリ（`skills/skill-name/`）として構成され、その直下に `SKILL.md` が配置されているか。
*   [ ] **フロントマターの必須項目**: `SKILL.md` の先頭にYAML形式で `name` と明確な発火条件を含む `description` が定義されているか。`name` の値とディレクトリ名が一致しているか。
*   [ ] **推論の排除**: `SKILL.md` のボディ部や `scripts/` に、AIによる複雑な「推論」「判断」「要件定義」を含んでいないか（神スキルになっていないか）。あくまで「外部システムの操作（副作用）や情報取得」に特化しているか。
*   [ ] **入出力の明確化**: スキルの使い方、引数（入力）、期待される出力結果、エラー時のハンドリングや自己修復のためのエラーメッセージがMarkdownボディ内に明確に記述されているか。
*   [ ] **段階的開示**: リファレンス等の詳細な情報がある場合、`SKILL.md` にすべて詰め込まずに `references/` などに分割してファイル参照させているか。

## 6. スキルのサンプルフォーマット

スキルディレクトリ (`read-file-content/`) を作成する場合の `SKILL.md` のサンプルです。

```markdown
---
name: read-file-content
description: 指定されたパスのファイル内容を読み込み、テキストとして返却します。ユーザーやエージェントがファイルの中身を確認する必要がある場合に【必ず】このスキルを使用してください。バイナリファイルには使用しないでください。
license: MIT
metadata:
  version: "1.0"
---

## 概要
このスキルは、プロジェクト内の特定のファイルの中身を読み取るために使用します。

## 使用手順
以下のPythonスクリプトを実行してファイルの内容を取得してください。

`scripts/read.py <filepath>`

### 入力
* `<filepath>`: 読み込む対象のファイルパス（リポジトリルートからの相対パス）。

### 出力とエラーハンドリング
* **成功時**: ファイルのテキスト内容が標準出力に返されます。
* **エラー時**: ファイルが存在しない、またはバイナリファイルである場合は、なぜ失敗したかの明確なエラーメッセージが返されます。エラーを受け取った場合は、パスが正しいか確認してリトライしてください。

## 詳細なリファレンス
特殊なエンコーディングのファイルについては、[エンコーディングリファレンス](references/encoding.md) を参照してください。
```
