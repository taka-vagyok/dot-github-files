# Skill Spec

本ドキュメントは、GitHub Copilot 向けのエージェントが実行するスキル (Skill/Tool) を作成する際のルールとベストプラクティスを定義したものです。

## 1. 明確な目的と粒度

*   **単一責務の原則**:
    1つのスキルは1つの明確なタスク（例: 「ファイルを読む」「ウェブページをスクレイピングする」「検索クエリを実行する」）に特化させるべきです。
*   **適切な抽象化**:
    あまりに細かすぎるスキル（例: 「1行だけ読む」）や、複雑すぎるスキル（例: 「ビルドしてテストしてデプロイする」）は避け、エージェントが組み合わせて使いやすい粒度で設計します。

## 2. 説明とメタデータ

*   **説明 (Description) への発火条件の明記**:
    スキルの `description` (説明文) はエージェントがそのスキルを選ぶための唯一の判断基準です。「いつ使うべきか」「何を行うか」「使用してはいけない場合はあるか」など、具体的な発火条件を含めて詳細に記述してください。
*   **直感的な名称**:
    ツール名は動詞と目的語を含めた簡潔で明確なもの (例: `read_file`, `search_github_issues`) を使用してください。

## 3. パラメータの定義

*   **包括的なパラメータ説明**:
    引数 (Parameters) を取る場合は、引数の型、必須かどうかのフラグ、および各引数の説明（例: 「ファイルパスのルートからの相対パス」「検索するキーワード文字列」）を正確に定義します。
*   **デフォルト値の活用**:
    頻繁に使われる設定があればデフォルト値を設け、必須パラメータの数を最小限に抑えることでエージェントが使いやすくなります。

## 4. エラーハンドリングと戻り値

*   **明確なエラーメッセージ**:
    スキルが失敗した場合は、エージェントが自己修復やリトライを行えるよう、「なぜ失敗したか」「次にどうすればよいか」を具体的に示すエラーメッセージを返却してください。
*   **一貫性のある戻り値**:
    戻り値はプレーンテキスト、JSON文字列など、パースしやすく一貫したフォーマットを採用します。
